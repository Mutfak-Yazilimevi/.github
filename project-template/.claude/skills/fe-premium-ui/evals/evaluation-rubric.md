# Evaluation Rubric

Score each dimension from 1 to 5.

## 1. Intent Mode Detection

- 1: Chooses the wrong mode.
- 3: Chooses a plausible mode but misses nuance.
- 5: Correctly detects Mode A or Mode B and explains why.

## 2. Project Type Identification

- 1: Generic or incorrect project type.
- 3: Identifies broad category.
- 5: Identifies specific UI type and its design implications.

## 3. UI Goal Understanding

- 1: Repeats user words without interpreting goal.
- 3: Understands surface goal.
- 5: Connects user goal to user action, product context, and emotional impression.

## 4. Avoids Vague Aesthetic Advice

- 1: Says "make it modern" or "add gradients."
- 3: Some concrete advice, some vague language.
- 5: Every aesthetic direction maps to layout, tokens, components, or states.

## 5. Information Architecture

- 1: No page structure.
- 3: Basic sections listed.
- 5: Clear hierarchy, CTA structure, navigation, and state planning.

## 6. Visual System

- 1: Random style words.
- 3: Basic palette and typography advice.
- 5: Concrete color, type, spacing, radius, shadow, border, asset, and data-viz direction.

## 7. Component System

- 1: Components missing.
- 3: Components listed without states.
- 5: Components include purpose, anatomy, states, interaction, and implementation notes.

## 8. Motion and Interaction

- 1: No motion guidance or excessive decoration.
- 3: Basic hover/transition suggestions.
- 5: Motion explains state, includes reduced motion, and says what not to animate.

## 9. Avoids Over-design and Trend Stacking

- 1: Adds many unrelated effects.
- 3: Mostly restrained but includes unnecessary trends.
- 5: Uses effects only when tied to product intent.

## 10. Frontend Implementation Guidance

- 1: Not implementable.
- 3: Mentions stack but lacks details.
- 5: Provides stack-adapted component, styling, state, accessibility, and responsive guidance.

## 11. Codex-ready Prompt

- 1: Prompt is vague.
- 3: Prompt is usable but incomplete.
- 5: Prompt includes context, structure, visual direction, components, motion, constraints, avoid list, and acceptance criteria.

## 12. Beginner-friendly Explanation

- 1: Jargon-heavy.
- 3: Some explanation.
- 5: Explains what to change, why it matters, and how to implement it in plain language.

## 13. Product Realism

- 1: Produces concept-shot advice.
- 3: Adds some realistic states.
- 5: Includes real workflows, realistic content, states, and responsive behavior.

## 14. Acceptance Criteria

- 1: None.
- 3: General criteria.
- 5: Testable criteria for hierarchy, components, states, responsiveness, and implementation.

## Overall Quality Bands

- 14-34: Weak. Mostly aesthetic phrasing.
- 35-52: Basic. Some structure but not implementation-ready.
- 53-64: Good. Useful for design planning.
- 65-70: Excellent. Ready for Codex implementation.
